### Issue #15: Custom AI Endpoint Timeouts (60s limit exceeded)

**рџ“… Date**: April 23, 2026
**рџЏ·пёЏ Category**: Backend / AI Pipeline
**рџљЁ Priority**: Critical

#### рџ“ќ Issue Description:
Custom AI providers (especially browser automation proxies or local Ollama instances) were timing out during tests and article processing. The system had a hardcoded 60s limit for tests and 120s for the pipeline.

#### рџ”Ќ Symptoms:
- Error message: "The operation was aborted due to timeout (60030ms)"
- AI Terminal tests failing even when the provider is reachable.
- Pipeline stages failing with "Connection failed" or timeout errors.

#### рџ’Ў Root Cause:
- **Tight Timeouts**: Modern AI models or proxies can take >60s to generate a response, especially under load.
- **Streaming by Default**: Some proxies return `data: ...` (SSE stream) by default, which can cause JSON parsing errors if not explicitly disabled.

#### рџ› пёЏ Applied Solution:
1. **Extended Timeouts**: Increased test timeout to 180s and pipeline timeout to 300s (5 minutes).
2. **Forced JSON**: Added `stream: false` to all OpenAI-compatible requests to ensure stable JSON responses.

#### вњ… Solution Verification:
- Manual `node` test confirmed the provider `apikey.4kmedialive.com` takes ~70s to respond.
- Extended timeouts allow these slow providers to return their status (even if they return an error from their own backend).


### Issue #14: Articles Created via RSS Pipeline Hidden from UI (Missing user_id)

**рџ“… Date**: April 20, 2026
**рџЏ·пёЏ Category**: Backend / Database / Pipeline
**рџљЁ Priority**: Critical

#### рџ“ќ Issue Description:
Articles automatically generated by the RSS pipeline did not appear in the "Articles" list or "Pipeline Status" boxes (Pending/Processing), even though the backend was actively processing them and logs were visible.

#### рџ”Ќ Symptoms:
- "Articles" page shows "0 articles total" or missing recent ones.
- "Pipeline" dashboard shows "0 Pending" and "0 Processing" even when "Running".
- Global "Logs" list correctly shows activity for "Article X".
- Pipeline Stage Performance bars are accurately reflecting work, but counters are zero.

#### рџ’Ў Root Cause:
1. **Missing Initialization**: In `artifacts/api-server/src/pipeline/index.ts`, the `runPipeline` function inserted new articles using `db.insert(articlesTable).values({...})` but failed to set the `user_id` field.
2. **Strict Filtering**: The API routes and counts queries (`GET /api/articles` and `GET /api/pipeline/status`) filter records using `WHERE user_id = ?`. Since the field was NULL, these articles were excluded from all user-specific views.
3. **Docker Propagation**: Initial code fixes in source files were not reflected in the running system because the API container was running a pre-built image without volume mapping for source code.

#### рџ› пёЏ Applied Solution:
1. **Code Fix**: Modified `runPipeline` in `pipeline/index.ts` to assign `site.user_id` to every article created from an RSS feed.
2. **Data Migration**: Executed direct SQL updates:
   ```sql
   UPDATE articles a SET user_id = s.user_id FROM sites s WHERE a.site_id = s.id AND a.user_id IS NULL;
   ```
3. **Container Rebuild**: Rebuilt the API service to activate the new code:
   ```bash
   docker-compose up -d --build api
   ```

#### вњ… Solution Verification:
- Database query confirms all articles now have the correct `user_id`.
- Articles now appear in the UI list and counters.

#### рџ“љ Lessons Learned:
- Always ensure every new database record is linked to a `user_id` if the UI/API relies on it for filtering.
- When working with Docker, source code changes require a container rebuild/restart unless dev-volumes are mapped.
- Always check the "owner" context when porting background pipeline logic.

### Issue #1: DB Schema вЂ” camelCase vs snake_case Mismatch

**рџ“… Date**: April 2026
**рџЏ·пёЏ Category**: Database / Backend
**рџљЁ Priority**: Critical

#### рџ“ќ Issue Description:
The original `imageupdatenew` project used Drizzle ORM with **camelCase** property names (e.g., `userId`, `siteId`, `rssUrl`). The `blogupdatenew` project's pipeline code queried columns using **snake_case** (e.g., `user_id`, `site_id`, `rss_url`). After merging both into a single DB schema, the pipeline either crashed or silently returned undefined for all joined queries.

#### рџ”Ќ Symptoms:
- Pipeline returned 0 articles even when sites existed
- `rssFeedsTable.userId` was `undefined` at runtime (column didn't exist in object)
- `INSERT` statements failed with "column does not exist" errors
- Drizzle query builder generated wrong SQL column names

#### рџ’Ў Root Cause:
Drizzle ORM uses the **TypeScript property name** in queries unless you explicitly set a SQL column name. The original tables were defined as:
```typescript
// WRONG вЂ” property name becomes SQL column name
userId: integer("userId")  // generates SQL: "userId" (case-sensitive, fails)
```
But the actual PostgreSQL columns were named `user_id`, `site_id` etc. (snake_case). After the merge, the new pipeline code used `rssFeedsTable.user_id` which didn't exist as a TS property.

#### рџ› пёЏ Applied Solution:
Rewrote all affected table schemas to use snake_case property names that match the actual DB column names:

```typescript
// CORRECT вЂ” property matches SQL column
export const rssFeedsTable = pgTable("rss_feeds", {
  id: serial("id").primaryKey(),
  user_id: integer("user_id").notNull().references(() => usersTable.id),
  site_id: integer("site_id").notNull().references(() => sitesTable.id),
  rss_url: text("rss_url").notNull(),
  is_active: boolean("is_active").default(true),
  // ...
});
```

Also ran `ALTER TABLE` statements to add missing columns:
```sql
ALTER TABLE articles ADD COLUMN IF NOT EXISTS source_url TEXT;
ALTER TABLE articles ADD COLUMN IF NOT EXISTS rank_math_score INTEGER;
ALTER TABLE pipeline_logs ADD COLUMN IF NOT EXISTS site_id INTEGER;
```

#### вњ… Solution Verification:
- Pipeline queries returned correct results
- `db.select().from(rssFeedsTable).where(eq(rssFeedsTable.user_id, id))` worked correctly

#### рџ“љ Lessons Learned:
- Always define Drizzle ORM property names to exactly match PostgreSQL column names
- When merging two projects, audit all schema files for naming convention mismatches before running any queries

### Issue #2: `settingsTable` Not Exported from DB Package

**рџ“… Date**: April 2026
**рџЏ·пёЏ Category**: Backend / Build
**рџљЁ Priority**: Critical

#### рџ“ќ Issue Description:
The pipeline code from `blogupdatenew` imported `settingsTable` from `@workspace/db`, but the unified platform only exported `systemSettingsTable`. This caused a build failure: `Module '@workspace/db' has no exported member 'settingsTable'`.

#### рџ› пёЏ Applied Solution:

```typescript

```

Also updated `lib/db/src/index.ts` to export the alias.

#### вњ… Solution Verification:
- Build succeeded: `вљЎ Done in 367ms`
- Pipeline imported `settingsTable` without errors

#### рџ“љ Lessons Learned:
- When merging projects, add compatibility aliases rather than doing a mass find-replace that can break other code
- Export aliases are zero-cost and prevent breaking downstream consumers

### Issue #13: Uploaded Template Images Disappearing (Ephemeral Docker Storage)

**рџ“… Date**: April 2026
**рџЏ·пёЏ Category**: Docker / Backend
**рџљЁ Priority**: High

#### рџ“ќ Issue Description:
When a user uploaded a custom background image or a photo layer in the Template Editor, it displayed correctly and the template saved successfully. However, upon reopening the template later, the images were missing (replaced by solid colors or wireframes).

#### рџ”Ќ Symptoms:
- Photo areas showed the default placeholder after reloading
- Custom background images failed to load
- Network tab showed `404 Not Found` for URLs like `/api/photo/file/1776645585317-***.png`

#### рџ’Ў Root Cause:
There were two independent problems causing this:
1. **Docker Ephemeral Storage**: The `api` server (`photo.ts` route) uploaded user images to the local `/app/artifacts/api-server/uploads` directory. However, this directory was not mapped to a persistent Docker Volume. Whenever the `api` container was restarted, the `uploads` folder was completely wiped out.
2. **Nginx Routing Precedence Bug**: Nginx had a regex configuration `location ~* \.(js|css|png|jpg...)$` which took higher priority than the `location /api/` proxy block. When the frontend requested `/api/photo/file/xyz.png`, Nginx incorrectly intercepted it because it ended in `.png` and tried to serve it as a static frontend file (yielding a 404), completely ignoring the Express endpoint!

#### рџ› пёЏ Applied Solution:
1. Modified `docker-compose.yml` to define and map a persistent volume named `uploads_data` to the API service to stop the data loss during restarts.
2. Modified `nginx.conf` and updated the `location /api/` block to use the `^~` modifier (`location ^~ /api/`). This tells Nginx that this prefix match has higher priority than regex matching, ensuring all requests starting with `/api/` are correctly proxied.

#### рџ“љ Lessons Learned:
- Never store stateful user uploads inside the container's local filesystem without mapping it to a Volume or using external object storage (like AWS S3).
- Always verify Docker volume mappings for any directory that handles persistent file uploads.

## рџ’ј Tips for New Developers

### Before making any change:
1. Read `replit.md` for the current architecture overview
2. Read this file to understand past mistakes
3. Check `lib/db/src/index.ts` exports before assuming a table is available

### File naming conventions:
- DB schema properties: **snake_case** (matches PostgreSQL column names)
